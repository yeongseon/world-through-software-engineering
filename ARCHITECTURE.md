이 문서는 프로젝트 구조에 대한 주요 결정과 그 근거를 기록합니다.

## ADR-001: 시리즈 기반 구조

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
초기에는 "시즌(season)" 기반으로 설계했으나, 시즌은 시간순 진행을 암시하고 독립적 도메인 확장에 부적합했다.

### 결정
**시리즈(series)** 기반으로 전환한다. 각 시리즈는 하나의 도메인 영역을 다루며, 독립적으로 읽고 확장할 수 있다.

### 근거
- 시리즈는 순서 없이 병렬 확장 가능
- Netflix 시리즈처럼 독립적이면서 같은 브랜드 하에 공존
- 새로운 도메인 추가가 기존 구조에 영향을 주지 않음

## ADR-002: 2-레이어 구조 (system + cases)

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
초기에는 "시스템 + 인시던트"로 설계했으나, 기업 도메인에서는 "인시던트"가 장애/사고만 의미하여 범위가 좁았다.

### 결정
모든 도메인에서 **system/** + **cases/**의 2-레이어 구조를 사용한다.

### 근거
- "케이스"는 장애뿐 아니라 조직 변화, 정책 전환, 성공적 실행 등도 포함
- 실제 엔지니어링 조직의 Design Doc + Postmortem/Case Study 구조와 일치
- system → cases 의존 관계가 명확 (케이스는 반드시 시스템 글에 의존)

## ADR-003: 계층형 용어 사전

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
단일 glossary.md 파일은 시리즈가 늘어날수록 비대해지고, 시리즈별 특화 용어와 공유 용어의 구분이 불가능했다.

### 결정
2-계층 glossary 구조를 사용한다:
- `docs/glossary/core.md` — 프로젝트 전체 공유 용어 (SSOT)
- `docs/series/{series}/glossary.md` — 시리즈 전용 용어

### 근거
- core glossary가 SSOT (Single Source of Truth)로 일관성 보장
- 시리즈별 glossary에서 도메인 특화 용어 관리
- 충돌 시 core가 우선 — 명확한 우선순위

## ADR-004: docs/topics/ 크로스 도메인 영역

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
일부 주제(예: "정부 예산 vs 기업 예산")는 여러 시리즈를 가로지르며, 특정 시리즈에 넣으면 다른 시리즈에서 찾기 어렵다.

### 결정
`docs/topics/` 디렉토리를 만들어 시리즈 횡단 주제의 canonical home으로 사용한다.

### 근거
- 시리즈 간 교차 분석의 명시적 공간
- 글의 canonical home이 모호해지는 문제 해결
- frontmatter의 `canonical_series` 필드로 원래 시리즈 표시 가능

## ADR-005: Article-as-Folder 패턴

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
글에 이미지, 다이어그램 등 에셋이 포함될 경우, 에셋 관리 방식이 필요했다.

### 결정
- 에셋 없는 글 → 단일 `.md` 파일
- 에셋 있는 글 → 폴더로 전환: `slug/index.md` + `slug/assets/`

### 근거
- 에셋 없는 글이 대부분이므로 기본은 단일 파일 (불필요한 복잡성 방지)
- 에셋이 생기면 폴더로 승격 — 점진적 복잡성
- git에서 rename으로 추적 가능

## ADR-006: 선택적 하위 폴더

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
하나의 도메인(예: politics/) 안에 system/과 cases/ 글이 많아지면 탐색이 어려워질 수 있다.

### 결정
- 기본: system/과 cases/ 안에 flat 구조
- 글이 ~10개를 초과하면: 하위 폴더로 그룹화 허용

### 근거
- 초기에는 flat 구조가 가장 간단
- 성장에 따라 자연스럽게 폴더 구조 도입
- 미리 하위 폴더를 만들면 빈 디렉토리만 늘어남

## ADR-007: 프론트매터 메타데이터 표준

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
모든 글에 일관된 메타데이터가 필요하며, 검색/필터/상태 관리의 기반이 된다.

### 결정
YAML frontmatter 표준을 정의한다:
- **공통**: article_type, id, title_ko, title_en, series, domain, status, last_reviewed, related, sources
- **system 전용**: notable_cases
- **case 전용**: primary_system, affected_systems, severity, tempo, impact_scope, failure_pattern

### 근거
- YAML frontmatter는 Markdown 생태계에서 표준
- 정적 사이트 생성기, 검색 도구와 호환
- 글 상태 5단계: draft → review → active → outdated → archived

## ADR-008: CC BY-SA 4.0 라이선스 + CLA

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
오픈소스 콘텐츠 프로젝트로서 적절한 라이선스가 필요하며, 향후 상업적 출판 가능성도 고려해야 했다.

### 결정
- 오픈소스 라이선스: **CC BY-SA 4.0**
- 기여자 동의: **CLA (Contributor License Agreement)** 적용

### 근거
- CC BY-SA 4.0: 자유로운 공유/수정 + 동일 라이선스 의무 → 커뮤니티 기여 유도
- CLA: 외부 기여분에 대해 상업적 활용 권한 확보 → 출판 시 법적 문제 방지
- 저작권자(프로젝트 소유자)는 별도 라이선스 부여 가능

## ADR-009: 모노레포 유지

**상태**: 확정  
**일자**: 2026-04-05

### 맥락
시리즈가 늘어날수록 레포를 분리할지 논의가 필요했다.

### 결정
시리즈 수가 5~7개가 될 때까지 **모노레포**를 유지한다.

### 근거
- 현재 2개 시리즈에서 레포 분리는 과잉 엔지니어링
- glossary, templates, docs/ 공유 자원이 모노레포에서 관리하기 쉬움
- 분리 기준: 시리즈별 기여자 그룹이 독립적으로 분화될 때
- GitHub org 전환도 이 시점에 함께 고려
